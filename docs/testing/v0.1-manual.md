# oby v0.1 — manual functional test prompts

A paste-into-claude test suite for verifying the v0.1 pipeline end-to-end. Each scenario is a self-contained prompt you copy into a wrapped Claude session, then visually check the activity feed against the expected behavior.

## How to use

1. In one terminal: `oby claude` (this opens a wrapped Claude session).
2. Paste the **Prompt** block for a scenario.
3. While Claude is working, press **Ctrl-G** to switch to the feed.
4. Compare what you see against the **Expected** block.
5. Press **Ctrl-G** to return to Claude. Press **q** in the feed to quit oby.

Between scenarios, you can keep the same session or restart with `oby claude` for a clean buffer. The ring buffer holds 500 entries per agent.

In the feed:
- `▸` = pending, `✓` = ok, `✗` = error.
- `←/→` switches between agents.
- Live byte chunks (the recovered stderr or stdout) appear under their entry, tagged `[stream-name]`.

## 0. Smoke test (run this first)

Verifies the full pipeline: hook activation → rewrite → tee → socket → ring → render.

**Prompt:**

```
Please do these three things in order, with no preamble:
1. Use the Read tool to read /etc/hostname.
2. Use the Bash tool to run: ls /nonexistent_dir 2>/dev/null; echo done
3. Stop and wait.
```

**Expected in feed:**

- A `▸ Read  /etc/hostname` entry. **Stays `▸` pending** — see Known limitations §1, CC doesn't fire PostToolUse for Read.
- A `▸ Bash  ls /nonexistent_dir 2>/dev/null; echo done` entry.
  - Underneath, a `[stderr-discarded]` chunk containing the real `ls: cannot access '/nonexistent_dir': No such file or directory` (which Claude itself never saw because of `2>/dev/null`).
  - A `[stdout]` chunk containing `done`.
- Both entries marked `✓` (the bash exit code is irrelevant here — the *capture* succeeded).

**If `[stderr-discarded]` is missing:** the rewrite did not engage. Check `jq '.hooks' ~/.claude/settings.json` to verify `oby install` ran, and that the wrapped claude session inherited `OBS_ACTIVE=1` / `OBS_SOCKET_DIR=...`.

## 1. Bash capturer — basic

**Prompt:**

```
Use the Bash tool to run: echo hello world
```

**Expected:**

- One `✓ Bash  echo hello world` entry on the `main` agent.
- A `[stdout]` chunk containing `hello world`.

**Verifies:** Bash capturer renders headline, status transitions Pending→Ok, stdout is teed.

## 2. Bash capturer — failed command

**Prompt:**

```
Use the Bash tool to run: false; echo $?
```

**Expected:**

- `✓ Bash  false; echo $?` (the *tool call* succeeded — exit code reaches Claude as part of the tool result; the entry shows ok unless the hook itself failed).
- `[stdout]` chunk: `1`.

**Verifies:** Non-zero exit does not break the capture pipeline.

## 3. Read capturer

**Prompt:**

```
Use the Read tool to read /etc/passwd. Just read it, don't show me the contents.
```

**Expected:**

- `▸ Read  /etc/passwd` entry on `main`. **Stays `▸` pending in v0.1** — CC doesn't fire PostToolUse for Read in 2.1.x, so the wrapper never sees the read complete.
- No `[stderr-discarded]` chunk (Read doesn't rewrite).

**Verifies:** ReadCapturer's `render_pre` fires and the entry routes correctly. The PostToolUse path is unverifiable until CC starts firing the event.

## 4. Discard recovery — `2>/dev/null` only

This is the headline v0.1 feature. The agent runs a command whose stderr is silenced, but oby recovers it.

**Prompt:**

```
Use the Bash tool to run: cat /etc/shadow 2>/dev/null; echo "i don't care if it failed"
```

**Expected:**

- `✓ Bash  cat /etc/shadow 2>/dev/null; echo "i don't care if it failed"`.
- `[stderr-discarded]` chunk: `cat: /etc/shadow: Permission denied` (or similar — whatever the OS actually printed).
- `[stdout]` chunk: `i don't care if it failed`.

**Verifies:** The `2>/dev/null` neutralization in `pre_rewrite` redirects the would-be-discarded stderr into oby-tee instead of `/dev/null`, while Claude still sees the empty-stderr tool result.

## 5. Discard recovery — quoted `2>/dev/null` should NOT be rewritten

The scanner must skip strings inside quotes.

**Prompt:**

```
Use the Bash tool to run: echo "this contains 2>/dev/null as text"
```

**Expected:**

- `✓ Bash  echo "this contains 2>/dev/null as text"`.
- `[stdout]` chunk: `this contains 2>/dev/null as text`.
- **No `[stderr-discarded]` chunk** — there was no actual redirect.

**Verifies:** The shell-aware scanner respects quotes.

## 7. Per-agent routing — single subagent

**Prompt:**

```
Use the Task tool to dispatch a general-purpose subagent. Tell it: "Run the Bash tool with command `echo from-subagent` and report back what you saw."
```

**Expected:**

- `main` agent: a `Task` tool entry (Task is not yet a capturer in v0.1 — it may not render an entry at all; that's fine).
- A **new agent** appears in the feed picker (bottom of the screen). Use `→` to switch to it. The label includes its agent_type (e.g., `[general-purpose]`).
- On the subagent's stream: `✓ Bash  echo from-subagent` with `[stdout]` chunk `from-subagent`.

**Verifies:** `agent_id` in the hook payload routes the subagent's commands to a separate ring buffer; the picker surfaces it.

**If both commands appear on `main`:** the hook isn't extracting `agent_id`, or the routing in the listener is broken. Check `crates/oby-core/src/hook.rs` (`HookContext::agent_key`).

## 8. Per-agent routing — concurrent subagents

**Prompt:**

```
Use the Task tool TWICE in parallel (in a single response with two tool calls) to dispatch two general-purpose subagents.

- Subagent 1: Run the Bash tool: echo "from agent A"
- Subagent 2: Run the Bash tool: echo "from agent B"
```

**Expected:**

- Two new agents appear in the picker, distinct from `main`. They get distinct `agent_key`s (e.g., synthesized from `agent_id`).
- Each subagent's feed contains exactly its own bash entry with its own stdout — no cross-contamination.
- Switching between them with `←/→` shows the correct entries on each side.

**Verifies:** Parallel dispatch routes correctly. No interleaved bytes across agents.

## 9. Volume — many quick commands

Tests that the ring buffer handles bursts and the feed renders smoothly.

**Prompt:**

```
Use the Bash tool to run this exactly: for i in $(seq 1 50); do echo "line $i"; done
```

**Expected:**

- One `✓ Bash` entry.
- `[stdout]` chunk with 50 lines.
- Feed renders without flicker or dropped chunks.

**Verifies:** ring buffer and live-chunk append handle bursts without dropping bytes (well below the 500-entry ring cap; this is volume per entry, not entry count).

Tests entry-count handling.

## 13. Hotkey & terminal handling

These don't involve Claude — just exercise the wrapper's terminal management.

**Inside an `oby claude` session:**

1. **Toggle once:** press `Ctrl-G`. Feed appears, no Claude residue underneath.
2. **Toggle back:** press `Ctrl-G`. Claude repaints fully (no blank screen). Cursor is back at the prompt.
3. **Toggle repeatedly:** press `Ctrl-G` ~6 times quickly. No visible corruption either way.
4. **Resize terminal while in feed view:** drag the window edge. Press `Ctrl-G` to return to Claude. Claude should redraw at the new size.
5. **Resize terminal while in Claude view:** drag the window edge. Claude redraws at new size.
6. **Quit from feed:** press `Ctrl-G` to enter feed, then `q`. oby exits, terminal is restored cleanly (try `ls` afterwards — should work, no raw mode residue).
7. **Crash exit:** in another terminal, `kill -9 <oby pid>`. Verify terminal is **broken** (this is a known limitation — the SIGKILL bypasses the TerminalGuard drop). Run `reset` to restore. *This is intentional v0.1 behavior; SIGKILL recovery would require a SIGTERM handler.*

**Verifies:** TerminalGuard restores raw mode + alt-screen on normal exit. The shrink-and-restore pty resize trick wakes Claude's redraw on every flip-back.

## 14. Off-pipeline session — hook self-disables

**Outside oby**, in a plain shell:

1. Run `claude` directly (not `oby claude`).
2. Make Claude run a Bash command.
3. Check that there is no error from the hook (it should env-gate itself and exit 0).

**Verifies:** `OBS_ACTIVE` env-gate works; oby-hook fails open when not in a wrapped session.

**How to check the hook ran silently:** the bash tool result should be unchanged compared to a control-group `claude` call without oby installed.

## Known v0.1 limitations (expected to behave incorrectly)

These are out of scope for v0.1 and will work in v0.2. Don't file them as bugs:

1. **Read entries stay `▸` pending and never reach `✓` or `✗`.** Empirically verified: CC 2.1.x doesn't fire `PostToolUse` for the Read tool (confirmed by installing a sentinel `tee` hook on PostToolUse Read — it never fires). The docs imply it should; this is a CC limitation, not an oby bug. The Read entry still records the path, just not the completion status.
- `cmd | grep …` — the discarded stdout from the upstream side of a pipe is NOT captured (only `2>/dev/null` is rewritten in v0.1).
- `cmd | head -n 5` — same as above.
- `cmd > FILE` — file redirection isn't intercepted.
- Multi-statement scripts don't get xtrace (`set -x` / `BASH_XTRACEFD`) — you only see the *outputs*, not the *commands* that produced them.
- Edit / Write / Grep / Glob / Task / WebFetch capturers don't exist yet — these tool calls will not show entries in the feed.
- Scrollback in the feed is unimplemented (the list auto-pins to the bottom).
- No config file — hotkey is hardcoded to Ctrl-G, ring capacity to 500.

## When something fails

A few useful checks:

```bash
# Verify hook is installed and points at the right binary.
jq '.hooks' ~/.claude/settings.json

# Verify oby-hook is on PATH.
which oby-hook

# Verify all four binaries are the same build.
which oby oby-hook oby-tee

# Look at the socket dir while a session is running.
ls -la $XDG_RUNTIME_DIR/obi/  # or /tmp/obi/ on macOS without XDG_RUNTIME_DIR

# Run oby-hook manually to see if it crashes on a known-good payload.
echo '{"hook_event_name":"PreToolUse","tool_name":"Bash","tool_input":{"command":"echo hi"},"tool_use_id":"t_1","session_id":"s","cwd":"/"}' | OBS_ACTIVE=1 OBS_SOCKET_DIR=/tmp/obi/dbg oby-hook
```

If a Bash command's `[stderr-discarded]` chunk is missing but the entry is there: the hook ran but the rewrite didn't engage, or `oby-tee` failed to connect. Check the socket dir.

If the entry itself is missing: the hook didn't run. Check the settings.json wiring and that the wrapped claude inherited the env.
