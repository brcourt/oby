# oby v0.2 — Design

**Goal:** Complete the Bash discard-recovery story that v0.1 only started. The wrapper currently catches `2>/dev/null` discards and that's it. v0.2 extends the rewrite engine to cover the other common discard patterns (`| grep`, `| head`, `| tail`, `> FILE`, `>> FILE`) and adds execution tracing (`set -x` via `BASH_XTRACEFD`) so multi-statement scripts show which command produced each output. Plus a small batch of polish items (`oby-hook --version`, nesting guard, pinned release artifact).

**Non-goals for v0.2:** new capturers (Edit/Write/Grep/Glob/Task/WebFetch), TOML config, single `oby` umbrella crate, search/filter in the feed, mouse mode toggle. Those land in v0.3+.

## 1. Scanner upgrade

v0.1's scanner in `crates/oby-hook/src/capturers/bash/scanner.rs` produces `Vec<CodeSpan>` — byte ranges of the command that aren't inside a quoted string or after a `#` comment. That's enough to find `2>/dev/null` safely; it isn't enough to know what's at a pipe boundary or what file a `>` points at.

v0.2 extends the scanner to additionally emit:

- **`Vec<PipeSegment>`**: byte ranges between unquoted `|` (single-pipe, NOT `||`) boundaries that are also outside subshells / command substitution / backticks. Each segment carries `{ start, end, first_word_start, first_word_end }` — the first whitespace-delimited token of the segment, which identifies the command (`grep`, `head`, `tail`, `awk`, …).
- **`Vec<Redirect>`**: positions of `>` and `>>` outside the same exclusion contexts, plus their right-hand target (a single word, optionally quoted, ending at whitespace or another shell metachar).

**Exclusion contexts the scanner must track:** single quotes, double quotes (with `\` escapes), `#` comments (already in v0.1), parenthesized subshells `( … )`, command substitution `$( … )`, backticks `` ` … ` ``, and (defensively) `[[ … ]]` test expressions. Nesting must work — `$( cmd $(inner) )` is one substitution containing another.

**Edge cases v0.2 explicitly does NOT handle** (documented as known limitations):

- Here-docs / here-strings (`<<EOF`, `<<<STR`).
- `&>` and `&>>` bash redirects (combine stdout+stderr).
- Process substitution as input (`< <(cmd)`).
- `coproc`, `select`, advanced bash control flow.

When a known-unhandled construct is detected, the scanner emits no pipe/redirect markers inside it. The rewriter falls back to passthrough for that region — command runs unchanged, no rewrite, no chunks captured. Safe by construction; user just loses observability for that specific invocation.

The scanner is hand-rolled, not a parser-crate dependency. Reasoning is captured in the v0.2 brainstorm: our needle-in-haystack job suits a 400-line byte-offset scanner better than a 5KLOC AST parser. v0.3+ reassesses if we hit a real bug whose root cause is "the scanner doesn't understand bash syntax."

## 2. Inner-pattern rewrites

Five new rewrite rules layered on top of v0.1's `2>/dev/null` neutralization. The outer wrap (the `{ … ;} > >(tee >(oby-tee stdout …)) 2> >(tee >(oby-tee stderr …) >&2)` envelope) stays exactly as v0.1.

| Source pattern in any pipe segment | Rewritten in-place to | New chunk stream label |
|---|---|---|
| `cmd \| grep PAT` (incl. `-v`, `-i`, `-E`, `-F`, `-w`) | `cmd \| tee >(oby-tee --stream stdout-pre-grep …) \| grep PAT` | `stdout-pre-grep` |
| `cmd \| head -n N` (incl. `head` w/o args, `head -c`) | `cmd \| tee >(oby-tee --stream stdout-pre-head …) \| head -n N` | `stdout-pre-head` |
| `cmd \| tail -n N` (incl. `tail` w/o args, `tail -c`, `tail -f` — see note) | `cmd \| tee >(oby-tee --stream stdout-pre-tail …) \| tail -n N` | `stdout-pre-tail` |
| `cmd > FILE` | `cmd > >(tee FILE >(oby-tee --stream stdout-to-file … >/dev/null))` | `stdout-to-file` |
| `cmd >> FILE` | `cmd > >(tee -a FILE >(oby-tee --stream stdout-appended-file … >/dev/null))` | `stdout-appended-file` |

**Semantics:** every rewrite captures the **upstream-of-filter** bytes, not the discarded subset. For `| grep PAT` the chunk shows the full producer output; the user reads "claude saw the matching lines from this larger producer output." For `> FILE` the chunk shows what got written to disk. This is simpler and more consistent than computing the dropped subset per filter (would require inverting grep flags, computing tail-line offsets, etc.), and it gives the user strictly more information than CC saw.

**Multi-stage pipelines (`cmd | grep | head`):** v0.2 captures the FIRST discarding stage only. The pre-grep chunk shows producer output; the post-grep / pre-head boundary is not captured. Multi-stage capture is v0.3 scope.

**Multi-redirect on one command (`cmd > FILE 2>&1`):** the `> FILE` rewrite still fires; `2>&1` is unchanged. Stderr ends up in FILE through bash's redirect-by-FD-merging, which the `tee` we injected preserves transparently.

**Note on `tail -f`:** infinite streaming. The `tee >(oby-tee --stream stdout-pre-tail …)` invocation captures bytes as they flow, indefinitely. oby-tee handles that fine — it streams to its socket and exits when stdin closes. No special handling needed; documented as expected behavior.

**Combined with `2>/dev/null` (v0.1):** both passes run in the same rewrite. v0.1's neutralization happens first; v0.2's pipe/redirect rewrites layer on top. They don't interact — `cmd 2>/dev/null | grep foo` becomes `cmd 2> >(oby-tee stderr-discarded …) | tee >(oby-tee stdout-pre-grep …) | grep foo` and the outer wrap captures the final stdout/stderr.

## 3. Execution tracing (BASH_XTRACEFD)

Bash's `set -x` echoes each command to a configurable FD before running it. v0.2 always enables xtrace inside the outer wrap and routes the trace to a third oby-tee invocation on a separate stream.

Outer wrap becomes (effectively, simplified):

```bash
export BASH_XTRACEFD=9
{
  set -x
  <user's command, possibly with v0.2 inner rewrites applied>
} > >(tee >(oby-tee --stream stdout …))   \
  2> >(tee >(oby-tee --stream stderr …) >&2) \
  9> >(oby-tee --stream xtrace … >/dev/null)
```

**FD 9 choice:** bash convention puts xtrace on a high FD to avoid clashing with normal 0/1/2 and with redirect-by-number constructs the user might use (`3>`, `4<`). 9 is unused in any realistic agent-generated script. Documented; if a collision ever surfaces, the scanner can detect `9>` / `9<` in the input and pick another high FD per invocation.

**`set -x` enabled for all Bash tool calls,** not just multi-statement ones — single-statement xtrace adds exactly one extra `[xtrace]` line that just re-prints the command. The consistency of behavior is worth that one line of redundancy. (Future v0.3+ config flag could disable per-invocation if it ever becomes annoying.)

**xtrace output format:** bash prints `+ <expanded command>` for each command at the current PS4 prefix. We don't override PS4; default `+ ` prefix is what users expect. Each traced command is a single line written by bash; oby-tee streams it as bytes; the TUI renders each line under the entry as a `[xtrace]` chunk just like other chunks.

**Combined with `set -e` / `pipefail`:** the user's command might set its own shell options. `set -x` is additive — it doesn't conflict. If the user's command does `set +x` partway through to silence tracing for a section, that's their call; we just stream whatever bash writes to FD 9.

**Failure mode:** if FD 9 redirection fails (extreme FD exhaustion in the inner bash), `set -x` falls back to stderr in bash's default behavior. That'd pollute the `stderr` chunk with `+ command` noise. Not a v0.2 concern — extreme FD exhaustion already breaks much more than this.

## 4. Polish

Three small items totaling ~100 lines of code + a CI workflow.

### 4a. `oby-hook --version`

The v0.1 hand-rolled main does no argv parsing. v0.2 adds a one-arg check before the tokio runtime spins up:

```rust
fn main() -> ExitCode {
    let mut args = std::env::args().skip(1);
    if matches!(args.next().as_deref(), Some("--version" | "-V")) {
        println!("oby-hook {}", env!("CARGO_PKG_VERSION"));
        return ExitCode::SUCCESS;
    }
    // existing tokio::main body, factored out into async_run()
    tokio::runtime::Runtime::new().unwrap().block_on(async_run())
}
```

Restores the strict version assertion in the brew formula test (`assert_match version.to_s, shell_output("#{bin}/oby-hook --version")`).

### 4b. Nesting guard

At the top of `oby/src/run.rs::run()` (the sync entry point, before any tokio init):

```rust
if std::env::var_os("OBS_ACTIVE").is_some() {
    eprintln!("oby: refusing to nest — OBS_ACTIVE is already set.");
    eprintln!("     you're trying to start an oby session from inside an existing one.");
    eprintln!("     exit this session first, or run plain `claude` for an unwrapped session.");
    return ExitCode::FAILURE;
}
```

This catches the "claude calls Bash with `claude` inside the wrapper" footgun the user hit during v0.1 testing. CC's Bash tool blocks on the nested interactive subprocess, the outer feed freezes, the terminal gets garbled because both claudes paint to the same alt-screen. With the guard, the nested invocation fails fast with a clear message and the outer session continues.

### 4c. Pinned release artifact

GitHub Actions workflow `.github/workflows/release.yml` that fires on `v*` tag push:

```yaml
on:
  push:
    tags: ['v*']

jobs:
  release-tarball:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - name: Build source tarball
        run: |
          TAG=${GITHUB_REF#refs/tags/}
          git archive --format=tar.gz --prefix="oby-${TAG#v}/" "$TAG" \
            > "oby-${TAG#v}.tar.gz"
      - name: Upload to release
        env: { GH_TOKEN: ${{ secrets.GITHUB_TOKEN }} }
        run: gh release upload "${GITHUB_REF#refs/tags/}" "oby-${TAG#v}.tar.gz"
```

The brew formula switches from `archive/refs/tags/v0.2.0.tar.gz` (which GitHub regenerates with potentially different gzip compression) to `releases/download/v0.2.0/oby-0.2.0.tar.gz` (an immutable release asset). Eliminates the SHA-rotation class of brew installation breaks that we hit during v0.1 setup.

## 5. Testing

### Unit tests per rewrite

Each new pattern in `crates/oby-hook/src/capturers/bash/rewrite.rs` gets at least:

1. **Positive case** — the canonical form (`cmd | grep foo`) rewrites correctly. Asserts the output contains `tee >(oby-tee --stream stdout-pre-grep` AND the original filter is preserved (`| grep foo` still present).
2. **Quoted-string negative** — `echo "ls | grep foo"` must not be rewritten. Asserts the output contains `"ls | grep foo"` literally.
3. **Comment negative** — `cmd # | grep foo` must not be rewritten.
4. **Within `$( … )` negative** — the pattern inside command substitution must not be rewritten as part of the outer command's pipe (the scanner sees `$( … )` as opaque).

For each of `grep`, `head`, `tail`, `>`, `>>`. Plus tests for the combined cases (`2>/dev/null` from v0.1 still works alongside).

### Scanner regression tests

`scanner.rs` gets new tests for:

- Subshell exclusion: `(echo a | grep b) | head -n 1` — only the outer `| head` is rewritable; the inner `| grep` is inside `(…)`.
- Command substitution exclusion: `echo "$(cmd | grep foo)"` — no rewrite anywhere.
- Backtick exclusion: same idea.
- Logical-or vs pipe: `cmd1 || cmd2 | grep foo` — only the rightmost `|` is the pipe; `||` is the logical-or.

### Xtrace integration test

A new test in `crates/oby-hook/tests/dispatch.rs` (or a new `tests/xtrace.rs`) that:

1. Spawns a fake oby-tee that accepts the connection and accumulates bytes.
2. Drives oby-hook with a Bash payload running a 3-statement script.
3. Asserts the xtrace stream received exactly 3 `+ <command>` lines.

### Manual smoke

`docs/testing/v0.2-manual.md` adds scenarios for each new pattern, mirroring the v0.1 testing doc structure. Each scenario gives a paste-into-claude prompt and the expected chunks to find in the oby feed. Existing v0.1 scenarios stay unchanged.

### Nesting guard test

Unit test in `oby/src/run.rs` (or wherever the guard lives) verifying that with `OBS_ACTIVE` set, `run()` returns `ExitCode::FAILURE` without attempting to spawn a pty.

## 6. Wire-format additions

`oby-core` declares a closed set of stream names today; v0.2 extends the set by convention only. No type changes are needed: `HeaderLine.stream` is already `String` (capturer-defined). The TUI's chunk rendering already iterates over `rec.live` and formats each as `[stream] line` — new stream names just appear with their new tag automatically.

No changes to `ControlMessage`, `DisplayEntry`, `DisplayEntryUpdate`, or the on-wire JSON envelope. Pure additive feature work.

## 7. Crate version bumps

All four crates go to 0.2.0. Publishes in the same dependency order as v0.1 (`oby-core` → `oby-tee` + `oby-hook` + `oby-cli`). Path deps in `oby-tee`, `oby-hook`, `oby-cli` get `version = "0.2.0"` for the path-to-`oby-core` dependency.

Brew formula updates `url` to the pinned release artifact URL, `sha256` to the v0.2.0 tarball's SHA, `version` block automatic from the URL.

## Known limitations carried over to v0.3+

- Multi-stage pipeline capture (`cmd | grep | head`) only sees the first stage.
- No capturers for Edit, Write, Grep, Glob, Task, WebFetch.
- No TOML config (hotkey, ring size, compose delay still env-var-only).
- No `cargo install oby` single-package install — still three crates, or homebrew.
- Here-docs, here-strings, `&>`, `coproc`, process substitution as input — not detected by the scanner; commands using them get no rewrites.
- Awk / sed / cut filters — not rewritten.
