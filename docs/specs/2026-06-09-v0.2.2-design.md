# oby v0.2.2 design — bounded pre-filter capture for grep, head, tail

**Goal:** Replace the unbounded "dump the entire upstream" behavior of `[stdout-pre-grep]`, `[stdout-pre-head]`, and `[stdout-pre-tail]` chunks with user-configurable context windows. The activity feed remains useful for discard-recovery without flooding it for large producer outputs.

**Non-goals for v0.2.2:** new capturers, TOML config, new shell support, awk/sed/cut filters, multi-stage pipeline capture beyond first filter. Those remain v0.3+. The `> FILE` / `>> FILE` redirect tees are out of scope here — their captured content is the file contents, which by definition is what the user wanted on disk; bounding makes no sense.

## 1. Motivation

v0.2.1 manual smoke testing on a 200-line `/etc/hosts` produced a `[stdout-pre-grep]` chunk containing all 200 lines plus surrounding rtk noise — useful for debugging but overwhelming for the agent activity feed's intended use as a quick "what did the agent's pipeline drop" reference. The signal-to-noise is worst on the most common case: `cat | grep PATTERN` against a large file.

Three pre-filter cases share the same shape problem:

| Filter | What claude sees | What activity feed shows today | Issue |
|---|---|---|---|
| `\| grep PAT` | matching lines | entire upstream | every non-matching line, all the time |
| `\| head -n N` | first N lines | entire upstream | every line past line N |
| `\| tail -n N` | last N lines | entire upstream | every line before the tail boundary |

For each, a small fixed-size context window is more informative than the dump: matched lines with surrounding context (grep), a peek past the truncation boundary (head), a peek before it (tail).

## 2. Architecture

Pure extension of v0.2.1. No new crates, no new dependencies. The `inject_pre_filter_tees` function in `crates/oby-hook/src/capturers/bash/rewrite.rs` is generalized: instead of emitting a fixed `tee >(oby-tee ... >/dev/null) |`, it now emits a per-filter wrap that runs the upstream bytes through a windowing pipeline before they reach oby-tee.

The user's original `| grep` / `| head` / `| tail` is unchanged — they still see exactly what they would have seen without oby.

The windowing pipeline runs in a separate process (via `>(...)` process substitution), so the agent's pipeline is not blocked waiting for windowing to complete.

## 3. Per-filter design

### 3.1 grep

**Today (v0.2.1):**

```
cat /etc/hosts | tee >(oby-tee --agent ... --stream stdout-pre-grep >/dev/null) | grep 127
```

**v0.2.2:**

```
cat /etc/hosts | tee >(grep -B 3 -A 3 127 | oby-tee --agent ... --stream stdout-pre-grep >/dev/null) | grep 127
```

The tee branch routes producer output through a **parallel grep** that reuses the user's pattern and flags verbatim, with `-B N -A M` prepended. The parallel grep does the windowing using grep's native context-printing — no custom matching logic. If the user's grep has any `-A`/`-B`/`-C` flags, grep's "last occurrence wins" semantics let user values override the prepended defaults (intentional: an explicit user value should always win).

**N and M values:**
- N = `$OBS_GREP_BEFORE_LINES` (default `3`)
- M = `$OBS_GREP_AFTER_LINES` (default `3`)

Read at PreToolUse hook invocation time. Embedded as literal numbers in the rewritten command — no env-var indirection in the shell because the values are deterministic at rewrite time.

**Skip condition — `-v` / `--invert-match`:**

When the user runs `grep -v PATTERN`, the lines the activity feed would show are the lines NOT matching the pattern — essentially the same as what claude sees. The pre-filter chunk would add noise without adding information. Detected via tokenized args; tee branch is omitted entirely for that segment (no chunk emitted).

Detection rules (best-effort, documented quirks):
- Standalone `-v` as a tokenized arg.
- Any short-flag run containing `v` (e.g., `-vi`, `-vE`, `-vIn`, `-vIE`).
- Long form `--invert-match` (exact match).
- NOT detected: a literal `-v` after `--`, or `-v` as the pattern itself (`grep -- -v file`). These are rare; documented as known limitations.

### 3.2 head

**Today (v0.2.1):**

```
cat huge.log | tee >(oby-tee --stream stdout-pre-head >/dev/null) | head -n 5
```

**v0.2.2:**

```
cat huge.log | tee >(tail -n +6 | head -n 3 | oby-tee --stream stdout-pre-head >/dev/null) | head -n 5
```

The tee branch does a **peek-ahead**: skip the first N lines (`tail -n +<N+1>`) and take the next M (`head -n <M>`). This gives M lines starting RIGHT AFTER the user's head boundary — the most useful context for "what would have come next."

**N parsed from user's head args:**
- `-n N` (POSIX): N is the explicit count.
- `-NUM` (BSD/GNU short form, e.g., `head -5`): N = the numeric value.
- No count: N = 10 (head's default).

**M:** `$OBS_HEAD_PEEK_LINES` (default `3`).

**Skip condition — `-c` (byte mode):**

`head -c N` truncates by bytes, not lines. Line-windowing makes no sense. When `-c` is detected:
- Emit no pre-head tee at all for that segment (skip the chunk).

Alternatives considered: fall back to current full-upstream capture. Rejected because byte-mode head is usually used on binary data where dumping the upstream as text is wrong anyway.

### 3.3 tail

**Today (v0.2.1):**

```
cat huge.log | tee >(oby-tee --stream stdout-pre-tail >/dev/null) | tail -n 5
```

**v0.2.2:**

```
cat huge.log | tee >(tail -n 8 | head -n 3 | oby-tee --stream stdout-pre-tail >/dev/null) | tail -n 5
```

The tee branch does a **peek-behind**: capture the last `(N + M)` lines via `tail -n <N+M>`, then take the FIRST M of those via `head -n M`. The result is the M lines RIGHT BEFORE the tail boundary — the context immediately preceding what claude sees.

**N parsed from user's tail args:**
- `-n N` (numeric POSIX form): N is the explicit count.
- `-NUM` (BSD/GNU short form): N = numeric value.
- No count: N = 10 (tail's default).

**M:** `$OBS_TAIL_PEEK_LINES` (default `3`).

**Skip conditions:**

1. **`-c` (byte mode):** as with head — skip.
2. **`+N` form (e.g., `tail -n +50`):** semantically different — shows from line N onward, not last N lines. The "peek-behind" idea doesn't translate cleanly. Skip; emit no pre-tail chunk.
3. **`-f` / `--follow`:** the user's pipeline runs indefinitely. Our windowing pipeline `tail -n <N+M> | head -n M` would deadlock waiting for tail to produce a finite stream. Skip.

## 4. Parsing rules

Common args parser used by head/tail (grep needs no count parsing since `-B`/`-A` are prepended to the original args). Implemented as a small private helper in `rewrite.rs`:

```rust
struct HeadTailArgs {
    count: usize,      // parsed N or default 10
    skip_windowing: bool,  // true if -c, -f (tail), or +N form detected
}

fn parse_head_args(args: &str) -> HeadTailArgs { ... }
fn parse_tail_args(args: &str) -> HeadTailArgs { ... }
```

Whitespace tokenization, no quote handling. Edge cases that produce wrong parses fall through to defaults; the worst case is showing 10 lines instead of N where N was a weirdly-formatted arg.

Approximate is OK because:
- If the parse is wrong, the WINDOWING pipeline still works — it just shows different lines than ideal.
- The user's actual head/tail is unchanged. No correctness issue, just chunk-content drift.

For grep, all we need to detect is `-v`/`--invert-match`. Same tokenization. No count extraction needed.

## 5. Environment variables

| Var | Default | Range | Used by |
|---|---|---|---|
| `OBS_GREP_BEFORE_LINES` | `3` | 0..=1000 | grep tee, prepended as `-B` |
| `OBS_GREP_AFTER_LINES` | `3` | 0..=1000 | grep tee, prepended as `-A` |
| `OBS_HEAD_PEEK_LINES` | `3` | 0..=1000 | head tee, peek count |
| `OBS_TAIL_PEEK_LINES` | `3` | 0..=1000 | tail tee, peek count |

- Read once via `std::env::var` at PreToolUse hook invocation.
- Parse failures or out-of-range values silently fall back to the default — no error to the user, no broken rewrite.
- Empty string or unset → default.
- Values > 1000 clamp to 1000 to prevent malicious or accidental denial-of-feed.

## 6. Rewrite shape changes

The `inject_pre_filter_tees` function (~30 lines today) grows to ~120 lines:

1. Walk each pipe segment as today.
2. For each segment after the first, look up the first word in a per-filter dispatch table:
   - `grep` → check skip condition (-v), build `grep -B N -A M ARGS` pipeline.
   - `head` → parse args, check skip (-c), build `tail -n +<N+1> | head -n M` pipeline.
   - `tail` → parse args, check skip (-c / -f / +N), build `tail -n <N+M> | head -n M` pipeline.
3. If skip condition is met for any filter, emit NO tee for that segment (the pre-filter chunk simply does not appear in the feed).
4. Otherwise, emit the tee with the windowing pipeline embedded.

The existing right-to-left `apply_inserts` infrastructure handles multi-stage pipelines like `cat | grep | head` correctly (both filters get windowed independently). The tee order is unchanged.

## 7. Wire format

No changes. `[stdout-pre-grep]`, `[stdout-pre-head]`, `[stdout-pre-tail]` chunk labels are the same. The TUI renders them identically. Only the **byte content** of those chunks changes — from full upstream to context windows.

## 8. Testing

### 8.1 Unit tests in `rewrite.rs`

For each filter, the following matrix:
- Default args, no env vars set → window with default values.
- Env var set to a non-default value → window uses that value.
- Skip condition triggered → no tee emitted (assert chunk absent in output).
- Multi-stage pipeline (`cat | grep X | head -n 5`) → both stages windowed independently.
- Existing v0.2.1 tests asserting on `[stdout-pre-grep]` / `[stdout-pre-head]` / `[stdout-pre-tail]` strings need updates: the wrap shape now embeds `grep -B 3 -A 3` / `tail -n + ... | head -n 3` etc.

### 8.2 Parsing tests

- `parse_head_args("")` → count=10, skip=false.
- `parse_head_args("-n 5")` → count=5, skip=false.
- `parse_head_args("-5")` → count=5, skip=false.
- `parse_head_args("-c 100")` → skip=true.
- `parse_tail_args("-n +50")` → skip=true.
- `parse_tail_args("-f")` → skip=true.

For grep:
- `grep_invert_match("PAT")` → false.
- `grep_invert_match("-v PAT")` → true.
- `grep_invert_match("-vi PAT")` → true.
- `grep_invert_match("--invert-match PAT")` → true.
- `grep_invert_match("PAT -v")` → true.

### 8.3 Integration test (optional, recommended)

Extend `crates/oby-hook/tests/xtrace.rs` or add `tests/pre_filter_context.rs`:

- Spawn fake oby-tee on PATH.
- Drive hook with `cat /tmp/seed.txt | grep PATTERN`.
- Verify the recorded `stdout-pre-grep.txt` contains only context-window lines, not the full seed.
- Repeat for head, tail, and each skip condition.

## 10. Version bump

v0.2.1 → v0.2.2. Patch release: env vars are opt-in (defaults preserve a meaningful behavior, just less verbose), no wire-format change, no API change. Existing PreToolUse hook installations keep working.

## 11. Known limitations carried over to v0.3+

- TOML config still deferred — env vars are the only knob in v0.2.2.
- Quoted args inside grep/head/tail invocations are tokenized naively; an arg like `'-v contained in a string'` would false-positive the -v skip detection. Acceptable: the worst case is a missing chunk.
- Process substitution and command substitution inside the grep pattern itself are not handled — the segment's pattern is copied verbatim into the parallel grep, which may behave differently if the pattern depends on environment that the tee branch doesn't have.
- The `tail -n +N` form is skipped rather than supported — peek-behind for that form would require knowing the line count, which is streaming-incompatible.
